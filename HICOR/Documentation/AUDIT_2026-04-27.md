# Audit Report — Phase 5 Implementation vs MIKE_RX_PROCEDURE.md

Scope: read‑only verification of `PrescriptionCalculator` and its dependencies against the source‑of‑truth procedure (last updated 2026‑04‑26). All file/line references are absolute to the repo as observed today.

---

## Step 1 — Sign determination + clinical gates (§3)

- **Implementation:** `PrescriptionCalculator.swift:277–316` (`appendClinicalGateFlags`)
- **Documented behavior (§3):** Even when 2 printouts agree, **require minimum 3 readings** when (a) antimetropia, (b) R/L SPH diff > 3.00, (c) one eye plano (±1.00) and other > ±5.00, (d) either SPH > ±10.00. Antimetropia stricter: **minimum 4 printouts**.
- **Actual behavior:** All four gates implemented as `ClinicalFlag.insufficientReadings(...)` advisories. Antimetropia uses `Constants.antimetropiaMinimumPrintouts = 4`; others compare against literal `< 3`.
- **Match:** ⚠ partial
- **Discrepancies:**
  1. Doc word is **"require"** — i.e. block computation. Code does **not** block: it computes the prescription anyway and just appends a flag. The orchestrator does not consult `insufficientReadings` when deciding `requiresManualReview` (which is hard‑coded `false` at `PrescriptionCalculator.swift:122`). UI is free to ignore the flag.
  2. Antimetropia detection here is `(r > 0 && l < 0) || (r < 0 && l > 0)` (strict, plano excluded). Same convention as `AnisometropiaChecker.isAntimetropia`. Consistent with the doc's "mixed sign" wording.

## Step 2 — Cross‑printout aggregation via Thibos M/J0/J45

- **Implementation:** `CrossPrintoutAggregator.swift:28–146`, with `PowerVector.swift:12–24`.
- **Documented behavior:** "Cross‑printout aggregation via Thibos M/J0/J45 power vectors" (Phase 5 step 2; §2 axis sliding scale; §10 SPH‑only).
- **Actual behavior:**
  - Computes **median** M, J0, J45 to seed outlier detection (lines 49–62).
  - Drops readings deviating from median by > 1.00 D (M), > 0.50 D (CYL), or > sliding‑scale axis tolerance (lines 67–111).
  - Recomputes **mean** M/J0/J45 on survivors and reconstructs `(sph, cyl, ax)` (lines 121–145).
  - SPH‑only readings: SPH contributes to M; CYL/AX excluded from vector math (lines 46, 95, 122, 152–154). ✓ §10.
  - Single‑reading short‑circuit at lines 35–44.
  - Degenerate "all flagged" fallback at lines 117–118 reverts to original readings with empty `dropped` list.
- **Match:** ✓ matches (with one caveat below)
- **Discrepancies:**
  - Math is correct; Thibos formulas in `PowerVector` are standard.
  - **Caveat:** When all readings are flagged, the dropped list is reset to `[]` so the operator sees zero exclusions even though the data was suspect. Likely benign — `ConsistencyValidator` should already have rejected such a session — but worth noting against the "every automatic exclusion is visible" principle (CLAUDE.md, `feedback_surface_automatic_exclusions.md`).

## Step 3 — Machine AVG validation (§4) — **the critical one**

- **Implementation:** `MachineAvgValidator.swift:20–37` and the call site in `PrescriptionCalculator.swift:151–204`.
- **Documented behavior (§4):**
  1. Default to machine AVG.
  2. Compute Thibos M from **raw readings**.
  3. If |machineAVG_M − computed_M| ≤ 0.50 → use machine AVG.
  4. If > 0.50 → outlier likely; apply MAD‑based outlier rejection on raw, recompute, **ignore machine AVG**.
  5. Mike's CYL caveat: if computed |CYL| > 1.00 → prefer the most‑negative raw SPH.
- **Actual behavior:**
  - `PrescriptionCalculator.computeEye` calls `CrossPrintoutAggregator.aggregate` **first** (which already drops outliers), gets `aggregate.sph/cyl/ax` and `computedM = PowerVector.toM(aggregate.sph, aggregate.cyl)`.
  - Then aggregates per‑printout machine AVGs via `aggregateMachineAvgs` (line 248) into a single synthetic AVG (mean SPH, mean CYL, Thibos‑averaged axis).
  - Calls `MachineAvgValidator.validate` comparing synthetic machine M to that already‑cleaned `computedM`.
  - If within 0.50 D → use machine AVG (`source = .machineAvgValidated`, lines 181–186).
  - Else → fall back to aggregate values, with the CYL caveat: when `|aggregate.cyl| > 1.00`, swap aggregated SPH for `eyeRaw.map(\.sph).min()` (most negative). Lines 187–204.
- **Match:** ⚠ partial — the I/O matches but the documented two‑stage architecture is inverted.
- **Discrepancies (critical issue):**
  1. **Order is reversed.** Doc: AVG‑vs‑raw‑M first → if disagree, drop outliers. Code: drop outliers first → compare AVG to filtered M.
     - Practical effect: in a 3‑printout scenario where one autorefractor printout itself is an outlier, the code drops that printout's raw readings during aggregation, then compares machine AVG (a mean of all 3 machine AVGs including the bad one) to the cleaned `computedM`. If the bad printout's machine AVG is far enough out, the synthetic AVG can be pulled away from the cleaned M and trigger fallback — **correct outcome by accident**.
     - In the opposite case where raw readings have a single bad reading INSIDE an otherwise good printout, that reading gets dropped before the AVG comparison; the code may then accept the machine AVG even though the doc's "validate computed‑from‑all‑raw" check would have flagged it.
  2. **"Aggregated machine AVG"** is a code construct not in the document. The doc treats machine AVG per printout. Code averages machine‑AVGs across printouts (mean SPH, mean CYL, Thibos‑averaged axis). Reasonable, but undocumented.
  3. **CYL caveat trigger:** doc says "if **computed** CYL > -1.00 D (more cylinder magnitude)". Code uses `abs(aggregate.cyl) > 1.00` — same threshold, but applies **only when machine AVG was rejected** (line 187 is in the `else` branch). The doc's wording is ambiguous about whether the caveat applies even when AVG is accepted. Flag for clarification.
  4. **CYL caveat picks raw SPH min only**, not raw CYL or raw AX. Doc only mentions SPH. Match.

## Step 4 — Outlier detection (MAD + 1.00 D / 0.50 D / sliding‑scale axis)

- **Implementation:** `CrossPrintoutAggregator.swift:67–111`; also a separate, AVG‑level outlier pass in `ConsistencyValidator.swift:194–264`.
- **Documented behavior (Phase 5 step 4):** "MAD‑based + 1.00 D SPH / 0.50 D CYL / sliding‑scale axis." `RESEARCH.md` per CLAUDE.md research item #2 specifies `k=3 MAD in power-vector space + 1.00 D ANSI floor`.
- **Actual behavior:**
  - Aggregator: deviation from **median** > fixed threshold (1.00 D / 0.50 D / `AxisMath.toleranceForCyl`). **No MAD scaling.**
  - ConsistencyValidator (`removeMajorityOutliers`): only fires at `count >= 3`; SPH outlier reason fires when `others.count ≥ 1` and `othersSpread ≤ 1.00 D` and `deviation > 1.00 D`. CYL reason requires `othersCyls.count ≥ 2`, so at exactly 3 printouts where one is SPH‑only, CYL outliers are not detected.
- **Match:** ⚠ partial
- **Discrepancies:**
  1. **MAD‑based component is missing.** No MAD calculation exists anywhere in `Services/Calculation/` or `Services/OCR/`. Implementation is fixed‑threshold deviation from median, not k=3 MAD. Likely fine on tight autorefractor data but the documented adaptive component is absent.
  2. **Asymmetric CYL outlier gate** in `ConsistencyValidator` (`>= 2` other CYLs) means SPH and CYL outliers have different sample‑size requirements at exactly 3 printouts. Whether intentional is unclear from doc.
  3. **Two outlier passes happen** — one upstream in `ConsistencyValidator` (AVG‑level, drops a whole eye section per printout) and one in `CrossPrintoutAggregator` (raw‑reading level, drops individual readings). Doc lists only one outlier step. Two passes is defensive but undocumented; the operator sees both via separate `upstreamDroppedOutliers` and `phase5DroppedOutliers` fields on the outcome, which the UI labels separately.

## Step 5 — Final value computation (rounding per §6)

- **Implementation:** `DiopterRounder.swift:18–67`.
- **Documented behavior (§6):**
  - SPH 0.25 D step. |CYL| > 1.00 → stronger; ≤ 1.00 → weaker.
  - CYL 0.50 D step. Tie direction per‑eye by |SPH| (≥ 3.00 → stronger; < 3.00 → weaker). Plano stays plano.
  - AX nearest integer degree.
- **Actual behavior:**
  - `roundSph`: ✓ matches. `stronger = abs(cyl) > 1.00`. Already‑on‑step short‑circuit at lines 27–30. Sign preserved at line 33.
  - `roundCyl`: ✓ matches. Plano short‑circuit at line 37. Lower/upper step distance compared with epsilon; tie resolved by `eyeSphMagnitude >= 3.00` (lines 50–58). The orchestrator passes `abs(roundedSph)` as the tie key (line 208) — **per‑eye, post‑SPH‑rounding**, matching doc examples.
  - `roundAx`: nearest integer, **clamped to [1, 180]**. Doc says "nearest integer degree" without specifying clamp; clamp is needed to satisfy `RawReading`'s 1–180 convention.
- **Match:** ✓ matches doc
- **Discrepancies:** none material. The 1–180 clamp is undocumented but consistent with the rest of the codebase.

## Step 6 — Anisometropia check (§8)

- **Implementation:** `AnisometropiaChecker.swift:27–51`, called at `PrescriptionCalculator.swift:54–68`.
- **Documented behavior (§8):**
  - Same‑sign: ≤ 2.00 normal; > 2.00 advisory; > 3.00 → take 3 readings, look for < 3 D option, otherwise refer out.
  - Mixed‑sign: both eyes within ±1.50 → dispense using **lowest absolute SPH**; either outside ±1.50 → refer out; minimum 4 printouts always.
- **Actual behavior:**
  - Same‑sign: `> 3.00` returns `sameSignReferOut` directly; `> 2.00` returns `sameSignAdvisory`. ✓ thresholds.
  - Antimetropia (mixed‑sign): if either eye |SPH| > 1.50 → `antimetropiaReferOut`; else `antimetropiaDispense(lowestAbsEye:)`.
  - 4‑printout gate handled in `appendClinicalGateFlags` as `insufficientReadings(...antimetropiaNeedsFour)`.
- **Match:** ⚠ partial
- **Discrepancies:**
  1. **"Take 3 readings, look for <3 D option" is not implemented.** Code goes from `> 3.00` straight to refer out. The doc describes a multi‑step procedure (escalate to 3 readings → re‑evaluate → only refer out if still > 3.00) that the code skips. May be intentional simplification or an oversight.
  2. **"Use the LOWEST absolute SPH value"** in the antimetropia‑dispense path: the code emits `.antimetropiaDispense(lowestAbsEye: lowest)` as a flag but **does not substitute that eye's SPH for the prescription**. Each eye's `FinalPrescription` carries its own computed SPH/CYL/AX. If the doc's intent is "apply the lower‑abs SPH value to both eyes" or "only dispense the lower‑abs eye," that's not happening. If the intent is just "operator awareness," the flag suffices. Doc wording is ambiguous; flag for clarification with Mike.

## Step 7 — Tier assignment (§7)

- **Implementation:** `TierAssigner.swift:18–51`, called at `PrescriptionCalculator.swift:81–98, 211`.
- **Documented behavior:**
  - Tier 0: BOTH eyes |SPH| ≤ 0.25 AND |CYL| ≤ 0.50 → symptom check.
  - Tier 1: SPH within ±6.00, CYL ≤ 2.00.
  - Tier 2: 6 < |SPH| ≤ 8 OR 2 < |CYL| ≤ 3 → patient notification.
  - Tier 3: |SPH| > 8 OR |CYL| > 3 → hard ceiling.
  - Tier 4: |SPH| > 12 → medical concern.
- **Actual behavior (`assignPerEyeTier`):**
  - Boundaries use strict `>` consistently. Order: Tier 4 → Tier 3 → Tier 2 → Tier 0 → Tier 1.
  - Tier 4 only fires on SPH (CYL never triggers Tier 4 — matches doc).
  - Per‑eye Tier 0 is computed; orchestrator only emits the symptom‑check flag when **both** eyes are Tier 0 (`PrescriptionCalculator.swift:96–99`). ✓ §7's "asymmetric Tier 0 dispenses with plano" rule.
  - Overall tier = max severity (`assignOverallTier` at line 37). Severity ranks Tier 0 lowest (0) so an asymmetric (Tier 0 / Tier 2) case surfaces as Tier 2 overall — correct per doc.
- **Match:** ✓ matches doc
- **Discrepancies:**
  - The **asymmetric‑Tier‑0 plano‑lens behavior** ("dispense plano for the low eye") is not enforced in the calculation — `FinalPrescription` for the Tier‑0 eye still carries its own (very low) SPH/CYL. The doc says "plano lens for the low eye"; that's a lens‑matching concern (Phase 6), not Phase 5, so this is likely out of scope here. Flagging since the doc lives under Phase 5 prose.

## Cross‑cutting observations

1. **`ConsistencyValidator` AVG‑level threshold:** `pairwiseAgreementCheck` uses `Constants.sphAgreementThreshold` (1.00) and `Constants.cylAgreementThreshold` (0.50) — matches §1. Sliding‑scale axis tolerance from `AxisMath.toleranceForCyl` — matches §2. Plano‑axis skip at lines 169–184 is a clinically sound undocumented refinement.

2. **`CLAUDE.md` is stale on ConsistencyValidator.** Its file responsibilities table says "per‑eye SPH/CYL spread > 0.75 D → `.inconsistentAddPhoto`". The actual implementation does AVG‑to‑AVG comparison with 1.00 D / 0.50 D thresholds. Code matches `MIKE_RX_PROCEDURE.md`; the description in `CLAUDE.md` is out of date.

3. **`requiresManualReview` is hard‑coded `false`** (`PrescriptionCalculator.swift:122`). Doc §5 specifies a Manual Review Required escalation path. Today that path is reached by `ConsistencyValidator` returning `.inconsistentEscalate(reason:)` — the `requiresManualReview` field on `PrescriptionCalculationOutcome` itself is never set true by Phase 5. Either the field is dead, or the orchestrator should be setting it under some condition (e.g., antimetropia + insufficient printouts, or "fewer than 2 agreeing across 4 printouts" — though that case never reaches Phase 5 today).

4. **§5 escalation conditions** ("4 printouts with fewer than 2 agreeing → Manual Review Required") are enforced upstream in `ConsistencyValidator.inconsistent(reason:count:)` via `count >= maxPrintoutsAllowed (5)` returning `.inconsistentEscalate`. Threshold is actually **5**, not 4 (`Constants.maxPrintoutsAllowed = 5`; `Constants.escalationPrintoutCount = 4` is defined but not referenced anywhere in the code). Doc §5 says "4 printouts" → escalate. **Code escalates only at 5**. Worth investigating whether this is a deliberate softening or a constant‑wiring bug.

5. **`PrescriptionSource` distinguishes** `.machineAvgValidated` / `.recomputedViaPowerVector` / `.recomputedWithOutliersDropped`. Each `FinalPrescription` carries this enum so the UI can attribute the source — clinical transparency.

## Summary

- **Steps fully matching documentation:** **5 of 9** (2, 5, 7, 8, 9).
- **Steps with discrepancies:** **4 of 9** (1, 3, 4, 6).
- **Critical issues** (potentially affect clinical output):
  1. **Machine AVG validation order is inverted** vs §4 (Step 3). End result usually similar but the documented "trust AVG, validate against raw, fall back if disagree" flow is actually "filter outliers first, then compare AVG to filtered M." Worth Mike review to confirm acceptable.
  2. **MAD‑based outlier rejection is not implemented** (Step 4). Fixed thresholds only. The k=3 MAD scaling promised in `RESEARCH.md` and Phase 5 step 4 is absent.
  3. **§5 escalation triggers at 5 printouts, not 4** (`Constants.escalationPrintoutCount = 4` is unused; `maxPrintoutsAllowed = 5` is what fires escalation).
  4. **Antimetropia "use LOWEST absolute SPH"** is emitted as a flag but not actually applied to the dispensed prescription values. Possibly correct (UI handles it) but doc is ambiguous.

     **RESOLVED April 28:** Mike confirmed each eye uses its own SPH; current code behavior (flag-only) is correct. Procedure document Section 8 updated to remove ambiguous wording.
  5. **Section 8 same‑sign refer‑out path** skips the doc's "take 3 readings, look for <3 D option" intermediate step.

- **Minor issues** (documentation/code drift, no clinical impact):
  - `CLAUDE.md` ConsistencyValidator description (0.75 D spread) is stale.
  - `axisFromVectors` plano‑axis convention (`180°` placeholder) is consistent across `PowerVector` / `CrossPrintoutAggregator` / `aggregateMachineAvgs` but not documented.
  - `PrescriptionCalculationOutcome.requiresManualReview` is dead today (always `false`).
  - `Constants.escalationPrintoutCount` defined but unused.
  - "Aggregated machine AVG" (mean of per‑printout AVGs) is a code construct not in the doc.
  - CYL outlier check in `ConsistencyValidator.removeMajorityOutliers` requires 2 other CYLs (asymmetric to SPH check).
  - "All readings flagged" fallback in `CrossPrintoutAggregator` resets `dropped` to `[]` — operator loses transparency in the rare degenerate case.

No code changes were made. Awaiting your prioritization on which discrepancies (if any) to address in follow‑up sessions.
